۲۸ ژانویه ۲۰۲۶، یک اپ شبکه‌اجتماعی به اسم Moltbook لانچ شد. بنیان‌گذارش با افتخار گفته بود حتی یک خط کد هم خودش نزده، کل پروژه رو با پرامپت زدن به هوش مصنوعی ساخته بود. سه روز بعد یکی دیتابیسش رو پیدا کرد کاملاً باز: ۱.۵ میلیون توکن API، ۳۵ هزار ایمیل، پسوردهای plaintext، همه روی یک کلید که طبق طراحی توی مرورگر همه کاربرا هم هست. کافی بود یک تنظیمات رو فعال کنن تا اون کلید از یک "پسورد مستر" به یک کلید بی‌خطر تبدیل بشه. فعالش نکردن.

چند ماه بعد، Lovable (یک شرکت ۶.۶ میلیارد دلاری دقیقاً توی همین کار) سورس‌کد و اطلاعات دیتابیس مشتری‌هاش رو **۴۸ روز** کامل در دسترس عموم گذاشت، چون یک روت API فقط چک می‌کرد کاربر لاگین کرده یا نه، نه اینکه آیا همون کاربری هست که حق دستکاری اون داده رو داره.

دو تیم متفاوت، دو محصول متفاوت، عین همون اشتباه احمقانه با یک کلاه دیگه. این اتفاق مدام تکرار می‌شه چون آدمایی که این اپ‌ها رو می‌سازن از هوش مصنوعی خواستن براشون کدش رو بزنه و هیچ دلیلی ندارن که بدونن "row-level security" چیه، یا چرا "لاگین کردن" و "اجازه‌ی این کار رو داشتن" دو سوال کاملاً جدا از همن. هر ابزار امنیتی توی بازار برای مهندسی نوشته شده که از قبل اینا رو می‌دونه. این یکی نه.

## برای کیه

یه چیزی رو vibe-code کردی (یه SaaS، یه ابزار، یه پروژه‌ی جانبی) بیشترش رو با حرف زدن با هوش مصنوعی ساختی. می‌خوای `vercel deploy` بزنی یا روی `main` پوش کنی، و یه حس ته دلت می‌گه قبلش یه چیزی رو باید چک کنی، فقط نمی‌دونی اون "یه چیزی" دقیقاً چیه. مخاطب این پروژه دقیقاً همینه. اگه از قبل کارت ریویوی امنیتیه، به این نیازی نداری. به [کاتالوگ چک‌ها](skills/seaworthy/reference/checks-catalog.md) نیاز داری برای خود پترن‌های regex، که پنج دقیقه‌ای می‌خونیش.

## واقعاً چیکار می‌کنه

کدت رو می‌خونه. هیچ‌وقت اجراش نمی‌کنه، به نت هم وصل نمی‌شه، و دنبال همون مشتی اشتباه مشخص می‌گرده که واقعاً اپ‌های واقعی رو ظرف چند روز از لانچ به فنا داده:

- یه API key یا پسورد واقعی که مستقیم توی فایل تایپ شده، یا یه `.env` که کامیت شده و بعد "حذف" شده (هنوزم توی تاریخچه‌ی گیتت هست)
- یه جدول Supabase که row-level security نداره و از طریق همون کلیدی که مرورگرت از قبل داره قابل دسترسیه
- کلید مستر دیتابیست، همونی که هر قانون دسترسی رو نادیده می‌گیره، یه جایی که مرورگر بتونه لودش کنه
- یه endpoint که چیزی رو آپدیت یا حذف می‌کنه بدون اینکه چک کنه درخواست‌دهنده اجازه‌ی این کار رو داره یا نه
- CORS کاملاً باز *و* قبول‌کننده‌ی کوکی، که یه مشکل متفاوت و بدتر از CORS باز به تنهاییه
- دیباگ‌مود که بی‌سروصدا توی چیزی که قراره نسخه‌ی پروداکشن باشه روشن مونده
- یه روت `/admin` که هرکسی فقط با گس زدن آدرسش می‌تونه بزنه
- یه storage bucket که هرکسی روی اینترنت می‌تونه توش فایل آپلود کنه

بعدش یکی از این سه تا رو بهت می‌گه:

```
✅ SAFE TO SHIP
⚠️  SHIP WITH CAUTION — 2 issues to fix first
🛑 DO NOT DEPLOY — 1 critical issue found
```

هر مشکل با فایل و لاین دقیقش میاد، با اینکه اگه نادیده‌ش بگیری دقیقاً چی به سرت میاد، و چقدر واقعاً طول می‌کشه درستش کنی. بدون حاشیه، و بدون اینکه با یه "با یه متخصص امنیت مشورت کن" از زیرش در بره، مثل خیلی از اسکنرهای دیگه.

## کجا به کارت میاد

دقیقاً قبل از هر دیپلوی، هر بار، عین چک کردن آینه‌ی ماشین قبل از خروج از پارکینگ. یه ترمینال توی پروژه‌ت باز کن و این رو بزن:

```
npx skills add AmirHosein-Lotfi/seaworthy@seaworthy && mkdir -p .claude/commands && curl -fsSL https://raw.githubusercontent.com/AmirHosein-Lotfi/Seaworthy/main/commands/sw.md -o .claude/commands/sw.md
```

همین، کل نصب همینه. هم خود اسکیل رو نصب می‌کنه، هم یه دستور `/sw` می‌سازه برای هرکسی که ترجیح می‌ده خودش صریح بگه به‌جای اینکه منتظر بمونه Claude خودش بفهمه (`/sw ./apps/web` هم کار می‌کنه اگه بخوای روی یه ساب‌فولدر بزنیش). در هر صورت، بازم می‌تونی بپرسی "این برای دیپلوی امنه؟" یا "می‌تونم این رو پوش کنم رو پروداکشن؟"، یا حتی نپرسی: طوری نوشته شده که خودش قبل از هر اتفاق دیپلوی-مانندی وارد عمل بشه. برای دیدن خروجی واقعی یه اسکن، [examples/sample-output-blocked.md](examples/sample-output-blocked.md) و [examples/sample-output-safe.md](examples/sample-output-safe.md) رو ببین.

روی ویندوز با PowerShell، این رو بزن:

```
npx skills add AmirHosein-Lotfi/seaworthy@seaworthy; New-Item -ItemType Directory -Force .claude/commands | Out-Null; Invoke-WebRequest https://raw.githubusercontent.com/AmirHosein-Lotfi/Seaworthy/main/commands/sw.md -OutFile .claude/commands/sw.md
```

## چیکار نمی‌کنه

این یه پن‌تست نیست، و یه پکیج npm آسیب‌پذیر رو پیدا نمی‌کنه، یا یه باگ منطق-کسب‌وکاری که سه قدم دورتر از پترن‌های بالاست، یا هرچیزی که نیاز به دیدن اجرای واقعی اپت داشته باشه. این همون چک پنج‌ثانیه‌ایه که می‌تونست دو تا حادثه‌ی واقعی و کاملاً قابل‌اجتناب رو متوقف کنه، نه چیزی فراتر از این، و این رو هم صریح ته هر گزارش می‌نویسه.

جزئیات کامل (هر چک، شدتش، و دستور دقیق برای اجرای دستی اگه دلت نمی‌خواد به یه اسکریپت اعتماد کنی) توی [reference/checks-catalog.md](skills/seaworthy/reference/checks-catalog.md) هست.

## از چی ساخته شده

خود تشخیص چندتا اسکریپت شل کوچیک و بی‌حاشیه‌ست (`skills/seaworthy/scripts/detectors/`) که هر کدوم دنبال یک چیز می‌گردن و یک خط JSON برای هر یافته چاپ می‌کنن. نه ripgrep، نه `jq`، هیچی بیشتر از `git` و ابزارهایی که هر ماشینی از قبل داره، و دقیقاً همین دلیلشه که این کار چند ثانیه‌ای تموم می‌شه. قضاوت‌های واقعی، مثل "این واقعاً یه کلید واقعیه یا فقط یه placeholder" یا "ممکنه این auth check یه wrapper سفارشی باشه که نمی‌شناسیم"، یه لایه بالاتر اتفاق می‌افتن، نه قایم‌شده توی یه regex. [reference/false-positive-rules.md](skills/seaworthy/reference/false-positive-rules.md) همه‌ی این قضاوت‌ها رو ثبت کرده، چون یه ابزار امنیتی که حتی یک‌بار اشتباه کنه، دیگه بار بعد کسی روش حساب نمی‌کنه.

چهار fixture تست پشت این قضیه‌ست: دوتا عمداً خراب (یکی بازسازی باگ Moltbook، یکی بازسازی باگ Lovable)، یکی با یه‌ذره از همه‌چیز، و یکی اپ تمیز که باید صفر یافته برگردونه. خودت اجراش کن:

```
bash tests/run-fixture-tests.sh
```

## برای چی ساخته شده

اول از همه Next.js و Supabase، چون دقیقاً استک پشت هر دو حادثه‌ی بالاست، به‌اضافه‌ی Node/Express معمولی. Firebase و Django/Flask یه شروع دارن ولی عمداً experimental علامت‌گذاری شدن، یعنی اعتماد کمتر تا وقتی از همون مسیر fixture که بقیه‌ی چک‌ها رد شدن، رد بشن. Rails، Laravel، موبایل، هرچی که داری ران می‌کنی: [CONTRIBUTING.md](CONTRIBUTING.md) دقیقاً می‌گه یه detector جدید به چی نیاز داره، و کوچیک‌تر از چیزیه که فکرش رو می‌کنی.
